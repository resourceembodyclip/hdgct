# Vokav Link Archive

Vokav Link Archive 是一个面向技术研究者与内容聚合者的结构化外链归档系统。该项目专注于对 vokav.cn 域名下散布于不同移动端子域（wap.mobile 与 h5.mobile）的深度文章链接进行集中式索引、分类与元数据提取，解决因子域名分散、URL 模式不统一导致的资源发现困难与引用追踪效率低下问题。项目定位为技术文档站点的外围基础设施，适用于需要批量处理移动端文章链接的自动化工作流。

本项目不对所收录链接的内容做任何修改或重写，仅提供链接的稳定化列表与基础元数据映射，确保原始数据完整性。当前批次为第 42/56 批，共包含 180 个资源链接。

## 功能概览

**链接去重与规范化校验**：自动检测同一文章 ID 在不同子域名下的重复出现，输出唯一性报告。

**子域名分类索引**：根据链接所属子域名（wap.mobile.vokav.cn 与 h5.mobile.vokav.cn）自动划分资源池，支持按子域名过滤导出。

**文章 ID 提取与排序**：从每条 URL 中解析出数字型文章 ID，并按 ID 数值升序或降序排列，便于增量更新追踪。

**元数据占位生成**：为每条链接生成可供后续填充的元数据模板，包括标题占位、文章类型推测、最后访问时间戳字段。

**批量健康检查接口**：提供对链接列表进行 HTTP 状态码批量探测的脚本入口，支持超时与重试策略配置。

**原始数据完整保留**：所有 URL 按照用户提供的原始字符串原样存储与展示，不进行协议升级、域名改写或路径规范化。

## 应用场景

技术文档站点的外链资产盘点：当文档站需要迁移或重构时，使用本项目的链接列表快速梳理所有外部引用来源，避免链接失效导致的文档质量下降。

移动端内容聚合平台的数据管道预处理：在内容入库前，利用本项目对分散在不同移动端子域的文章链接进行统一格式化与去重，降低后续 ETL 流程的异常处理成本。

学术研究中的引用网络构建：研究者可基于本项目的链接集合，进一步抓取页面标题、发布时间等元数据，构建小规模引用网络或话题热度分析数据集。

个人知识库的批量导入前处理：知识管理工具使用者可将本项目的输出列表作为输入，批量导入到个人笔记系统或书签管理工具中，实现跨设备访问。

## 快速开始

```bash
# 克隆仓库
git clone https://github.com/example/vokav-link-archive.git
cd vokav-link-archive

# 安装依赖（Python 3.8+ 环境）
pip install -r requirements.txt

# 运行链接索引脚本，生成 JSON 与 Markdown 双格式输出
python archive.py --input data/links_batch_42.txt --output output/batch_42
```

## 贡献指南

1. 在 GitHub 上 fork 本仓库并创建一个新的分支，分支命名遵循 `feature/描述` 或 `fix/描述` 格式，避免在主分支上直接提交。

2. 新增链接解析规则或修改现有提取逻辑时，请同步更新 tests 目录下的对应单元测试用例，确保所有测试通过后发起 Pull Request。

3. 提交信息请使用约定式提交规范（Conventional Commits），例如 `feat(parser): add support for new subdomain pattern` 或 `fix(dedup): correct ID comparison for large integers`。

4. 在提交前运行 `black` 与 `pytest` 进行代码格式化与全量测试，确保代码风格一致且无回归缺陷。

5. 若需要新增外部依赖，请同时在 requirements.txt 与 docs/developer_guide.md 中说明新增依赖的用途与版本约束。

## 常见问题

**问：项目是否会对链接内容进行抓取或存储？**

答：不会。本项目仅对链接字符串本身进行解析、去重与排序，不发起任何对目标服务器的请求，也不缓存或存储任何文章内容。健康检查功能为可选模块，默认关闭，需用户显式启用。

**问：如果同一个文章 ID 同时出现在 wap 和 h5 子域名下，如何处理？**

答：项目默认以文章 ID 为唯一键进行去重，保留首次出现的记录，并生成去重报告列出所有重复项的原始 URL。用户可通过配置参数选择保留哪个子域名的记录。

**问：输出的 Markdown 与 JSON 格式有何区别？**

答：Markdown 格式面向人类阅读，适用于直接插入文档或 Wiki；JSON 格式面向程序处理，包含更丰富的元数据字段（如解析时间戳、ID 类型标记等），便于后续自动化流水线集成。
