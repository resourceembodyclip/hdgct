# Vokav Mobile Article Gateway

Vokav Mobile Article Gateway 是一个面向移动端技术内容聚合与分发的高性能网关服务，专注于将分散的移动端技术文章、案例分析及工程实践通过统一的轻量级接口进行标准化输出。本项目面向移动端开发工程师、技术内容运营人员以及架构决策者，旨在解决移动端技术资料检索效率低下、内容源分散、格式不统一等核心问题，提供一套可私有化部署的技术内容索引与快速查阅方案。

本项目通过静态路由映射与动态内容预加载机制，将大量历史技术文档以结构化方式对外暴露，便于集成至企业内部知识库、自动化文档流水线或移动端调试辅助工具链中。项目本身不依赖重型数据库，基于文件系统索引与内存缓存，可在低功耗设备上稳定运行，尤其适合边缘计算节点或本地开发环境下的技术资料快速查阅。

## 功能概览

**静态路由自动注册**：系统启动时自动扫描指定目录下的 HTML 文档映射表，根据文件命名规则生成对应的访问路由，无需手动维护路由表。

**移动端自适应模板引擎**：内置轻量级模板渲染层，所有输出页面均针对移动端屏幕尺寸进行适配，支持主流 WebView 及移动浏览器内核。

**多级缓存与预加载策略**：对高频访问的技术文档实施内存与本地文件系统两级缓存，支持 LRU 淘汰策略，有效降低磁盘 I/O 开销。

**结构化元数据抽取**：从 HTML 文档头部及正文中自动抽取发布时间、技术标签、阅读时长、关联文章等元数据，形成结构化索引。

**全文检索与标签过滤**：基于倒排索引实现轻量级全文搜索功能，支持按技术领域、发布时间段、文章类型等多维度过滤。

**健康检查与流量监控**：提供内置的 /health 与 /metrics 端点，可对接 Prometheus 或 Datadog 等监控系统，实时追踪请求延迟与错误率。

**可插拔的鉴权中间件**：支持通过配置文件启用基于 Token 或 IP 白名单的访问控制，适用于企业内部受限内容分发场景。

**文档快照与版本回溯**：对每个技术文档保留最近三次修改的历史快照，支持通过 URL 参数快速回溯至指定版本。

## 应用场景

移动端研发团队内部知识库构建。团队可将过往积累的数百篇技术周报、踩坑记录、性能优化案例通过本项目统一汇聚，开发者在日常调试时可直接在移动设备上检索查阅，无需频繁切换至桌面浏览器或翻阅聊天记录。

自动化文档流水线的末端输出节点。在 CI/CD 流程中，每次构建完成后可自动生成当前版本的技术文档说明，并推送至本项目的指定目录，运维人员通过移动端即可实时查看最新构建对应的变更说明与技术参数。

技术峰会与黑客松现场的快速资料站。在缺乏稳定网络环境的活动现场，组织者可提前将全部演讲材料、实验手册导入本项目，参与者通过连接本地 Wi-Fi 即可在手机端访问所有资料，有效避免因网络拥堵导致的内容获取失败。

边缘计算节点上的辅助运维手册。在工业物联网或边缘网关部署场景中，本项目可运行在资源受限的 ARM 设备上，为现场维护人员提供设备调试命令集、常见故障处理流程等关键文档的移动端查阅入口。

## 快速开始

以下操作步骤演示了如何在 Linux 或 macOS 环境下从源码克隆、安装依赖并启动本项目。

```bash
git clone https://github.com/vokav/article-gateway.git
cd article-gateway
npm install
npm run build
cp .env.example .env
npm start
```

启动成功后，默认服务将监听在 3000 端口。您可通过浏览器访问 http://localhost:3000 确认服务状态。若需修改端口或调整缓存大小，请编辑 .env 文件中的相应参数后重启服务。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---------|---------|------|
| Node.js | 18.x 或 20.x LTS | 运行时环境，建议使用官方二进制分发版或 nvm 管理 |
| npm | 9.x 及以上 | 依赖包管理工具，随 Node.js 一同安装 |
| 文件系统 | ext4 / APFS / NTFS | 用于存储文档索引与静态资源，建议至少 512MB 可用空间 |
| 内存 | 512MB 及以上 | 生产环境建议 1GB 以上以启用完整缓存策略 |
| 操作系统 | Linux 内核 4.x+ / macOS 12+ / Windows 10+ | 支持主流 POSIX 兼容系统及 WSL2 环境 |
| 网络 | 无强制要求 | 若需远程拉取文档源，建议具备稳定的出站网络连接 |

## 常见问题

**问：服务启动后无法访问任何文档页面，返回 404 状态码，应如何排查？**

答：首先检查 data/index.json 文件是否存在且内容格式正确。该文件记录了所有可用文档的路由映射关系。若文件缺失，可运行 npm run seed 重新生成初始索引。其次检查日志输出中是否存在文件读取权限错误，确保运行服务的进程对 data/ 目录具有读取权限。若以上均正常，请确认请求 URL 中的路径部分是否与 index.json 中的 key 严格匹配，包括大小写与扩展名。

**问：内存缓存占用量持续增长，如何限制缓存大小或调整淘汰策略？**

答：在 .env 文件中设置 CACHE_MAX_SIZE 变量，单位为 MB，默认值为 256。项目使用 LRU 策略自动淘汰最久未访问的条目。若需调整淘汰算法为 LFU，可在 src/config.js 中修改 cacheStrategy 配置项，但需要重启服务生效。建议定期观察 /metrics 端点输出的缓存命中率，根据实际访问模式调整缓存容量。

**问：如何将本项目部署至 Nginx 或 Apache 之后，并启用 HTTPS？**

答：本项目本身设计为直接面向客户端提供服务，但也可置于反向代理之后。在 Nginx 配置中，需设置 proxy_pass 指向本地 3000 端口，并配置 proxy_set_header Host $host 以正确传递原始请求头。HTTPS 终端建议在反向代理层处理，项目内部保持 HTTP 即可，无需额外配置证书。若需在项目内部启用 TLS，可参考 docs/configuration.md 中的 SSL 相关参数，但生产环境更推荐由反向代理统一管理证书。
