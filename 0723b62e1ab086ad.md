# ResourceBridge 技术资源导航

ResourceBridge 是一个面向开发者和技术研究人员的结构化外部资源聚合与导航系统。本项目的核心定位并非内容生产，而是对互联网上分散的高质量技术文章、案例分析、开发笔记与工程实践进行系统性梳理、分类与索引，帮助技术从业者以最低的检索成本获取最相关的参考资料。项目目标用户包括后端工程师、前端开发者、运维人员、架构师以及技术团队的技术选型负责人。

本项目解决的核心问题是：技术信息碎片化导致的检索效率低下。在真实的研发流程中，开发者往往需要在多个平台间反复跳转、使用不同关键词进行多次搜索，才能拼凑出解决某个具体问题所需的完整上下文。ResourceBridge 通过预先定义的分类体系与持久化的外链索引，将这一过程从“每次重新检索”转变为“一次索引，多次复用”，从而显著降低团队内部及个人知识管理中的信息摩擦。

## 功能概览

**按技术领域分类索引** 系统按照编程语言、框架、基础设施、数据库、前端工程化等维度对资源进行标签化分类，方便按方向批量查阅。

**全文元数据提取** 对每个收录的外部链接提取标题、发布时间、内容摘要等元数据，形成可检索的资源卡片，而非简单的网址列表。

**批量导入与解析** 支持通过脚本或配置文件批量导入 URL 列表，自动完成去重、存活检测与基础信息抓取，适用于大规模资源迁移场景。

**多级标签体系** 允许为单条资源附加多个业务标签（如“高并发”、“分布式事务”、“CI/CD”），支持标签间的组合筛选与交集查询。

**资源状态监控** 定期对已收录的链接进行可用性检查，标注失效链接并生成报告，保证资源库的长期有效性。

**自定义分类视图** 用户可根据自身项目需要，将资源按自定义主题分组，并生成独立的导航页面或文档片段。

**命令行与 Web 双接口** 提供基于 CLI 的批量操作能力，同时提供轻量级 Web 界面供团队成员共享与浏览资源库。

**版本化变更追踪** 记录每次资源增删改的操作日志，支持回溯历史状态，便于多人协作时的变更审计。

## 应用场景

**技术团队新成员入职培训** 团队技术负责人可将本项目维护的资源库作为新员工的技术脉络参考，帮助其快速了解团队常用的技术栈、工具链以及典型问题的解决方案来源，缩短业务上手周期。

**个人技术博客的参考资料整理** 独立博主或技术写作者在撰写深度技术文章时，可将本项目作为外链素材的管理后台，集中存放文献引用、对比分析链接和案例源码地址，避免文章写作过程中频繁切换浏览器标签。

**微服务架构选型调研** 架构师在评估不同服务注册中心、API 网关或配置中心时，可将调研过程中查阅的官方文档、性能对比报告、社区讨论帖以及生产环境案例统一收录，形成结构化的选型报告依据。

**离线文档体系的补充索引** 对于部署在隔离网络环境下的内部文档站点，本项目的资源列表可作为外部互联网知识的“映射清单”，帮助运维人员在有限网络条件下规划需要提前缓存或镜像的关键外部依赖。

## 快速开始

以下步骤演示如何在本地环境部署 ResourceBridge 并导入第一批资源数据。

```bash
# 克隆项目仓库到本地
git clone https://github.com/resourcebridge/resourcebridge.git
cd resourcebridge

# 安装项目依赖（使用 Python 3.10 及以上版本）
pip install -r requirements.txt

# 初始化 SQLite 本地数据库并执行数据迁移
python manage.py migrate

# 运行内置的 URL 导入脚本，载入预设资源列表
python scripts/import_urls.py --source data/initial_urls.json

# 启动本地开发服务器，访问 Web 界面
python manage.py runserver --port 8080
```

首次启动完成后，可通过浏览器访问 http://localhost:8080 浏览已导入的资源条目，或通过命令行工具继续添加新的链接。

## 项目结构

```
resourcebridge/
├── cmd/                                 # 命令行入口与 CLI 命令定义
│   ├── server/                          # Web 服务启动入口 (main.go)
│   └── importer/                        # 批量导入工具的独立入口
├── internal/                            # 私有包，不对外暴露
│   ├── crawler/                         # HTTP 抓取与元数据提取核心逻辑
│   │   ├── fetcher.go                   # 发送请求、处理重定向与超时
│   │   └── parser.go                    # 基于 goquery 的 HTML 解析
│   ├── storage/                         # 数据库抽象层与迁移脚本
│   │   ├── sqlite.go                    # SQLite 驱动实现
│   │   └── migrations/                  # 版本化 DDL 变更文件
│   ├── scheduler/                       # 定时任务调度（资源存活检测）
│   │   ├── cron.go                      # 基于 cron 的调度配置
│   │   └── checker.go                   # 并发健康检查工作池
│   └── model/                           # 数据实体与业务模型定义
│       ├── resource.go                  # 资源条目结构体
│       └── tag.go                       # 标签与分类结构体
├── pkg/                                 # 可被外部项目引用的公共库
│   ├── httpclient/                      # 自定义 HTTP 客户端封装
│   └── logger/                          # 结构化日志工具（基于 zap）
```

## 贡献指南

1. 在 GitHub 上 fork 本仓库至个人账户，并 clone 到本地开发环境。建议在开发前阅读 `docs/developer/api-reference.md` 了解核心模块的设计约定。

2. 创建新的功能分支，分支名称遵循 `feature/` 或 `fix/` 前缀加简要描述，例如 `feature/add-tag-filter`。禁止直接在 main 分支上修改代码。

3. 完成代码修改后，运行 `make test` 确保所有已有单元测试通过，并为新增功能补充对应的测试用例（位于 `tests/unit/` 或 `tests/integration/`）。同时执行 `make fmt` 对代码进行统一格式化。

4. 提交变更时，使用约定式提交规范，提交信息格式为 `<type>(<scope>): <subject>`，例如 `feat(importer): support retry logic for failed fetches`。提交前请确保代码无静态检查告警。

5. 向主仓库的 main 分支发起 Pull Request，并在 PR 描述中清晰说明变更目的、影响范围以及测试验证情况。PR 需要至少一名项目维护者审核通过后方可合并。

## 常见问题

**Q：导入大量 URL 时出现超时或连接拒绝错误，如何解决？**

A：默认的 HTTP 客户端超时时间为 10 秒，对于响应较慢的外部站点可能不足。可以通过修改 `configs/config.dev.yaml` 中的 `crawler.timeout` 字段调整为 30 秒或更长。此外，部分站点可能对频繁请求实施访问限制，建议在 `internal/crawler/fetcher.go` 中启用 User-Agent 轮换和请求间隔延迟策略。

**Q：如何对已导入的资源进行批量重新分类或更新标签？**

A：可以使用 `scripts/batch_update_tags.py` 脚本，该脚本接受一个 CSV 文件，其中包含资源 ID 与新增标签列表的映射关系。运行命令为 `python scripts/batch_update_tags.py --mapping tags_update.csv --dry-run`，其中 `--dry-run` 参数可用于预览变更而不实际写入数据库。确认无误后移除该参数即可执行正式更新。

**Q：本地 Web 界面无法显示外部链接的标题和摘要，只显示 URL 本身，是什么原因？**

A：这通常说明元数据提取阶段未能成功抓取到目标页面的标题标签或 meta description。可能的原因包括：目标站点要求 JavaScript 渲染内容（此时需考虑更换为基于无头浏览器的抓取方案）、目标站点临时不可用、或网络环境存在代理限制。建议首先检查 `logs/crawler.log` 中的错误信息，确认具体失败原因。对于依赖 JS 渲染的站点，本项目当前版本暂不支持，可考虑将该类资源标记为“需手动补充元数据”。
