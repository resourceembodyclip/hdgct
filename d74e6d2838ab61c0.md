# LinkVault Core

LinkVault Core 是一个面向技术团队与内容运营者的轻量级外链资产归集与导航系统。该项目并非传统意义上的网页爬虫或链接检测工具，而是一个专注于对半结构化、非标准命名的 URL 资源进行批量提取、元数据增强、分类存储与快速检索的后端服务框架。其核心定位是解决在内容聚合、历史数据迁移或第三方接口对接过程中，大量不规则 URL 难以被有效管理和利用的问题。目标用户包括运维工程师、数据管道开发者、内容平台的后端研发人员以及需要处理大批量异构链接的技术文档维护者。

LinkVault Core 内置了灵活的规则引擎，能够通过可配置的路径模式匹配、参数清洗与去重策略，将原始链接转化为结构化的资源条目。项目不依赖外部搜索引擎或商业 API，完全基于正则表达式与本地缓存机制运行，适合部署在无公网环境的内网服务器或容器化平台中。通过提供一致的编程接口与命令行工具，该项目帮助用户将散落的文章、文档、媒体文件链接整合为可查询、可导出、可监控的统一资源目录。

## 功能概览

**批量链接导入** 支持从纯文本文件、CSV 或标准输入流中批量读入 URL 列表，自动识别协议头与域名归属，并对每一条原始链接进行格式校验与规范化处理。

**路径语义解析** 针对形如 /Article/details/{id}.sHtML 的路径结构，提供基于正则捕获组的动态分段提取，能够将文章 ID、分类标识、文件扩展名等关键字段拆解为独立数据列，便于后续筛选与排序。

**元数据补充机制** 允许用户通过外部字典文件或数据库查询结果为每个链接补充自定义标签、备注、状态标记以及优先级分数，从而在原始 URL 基础上构建更丰富的业务上下文。

**去重与冲突检测** 内置基于 URL 完整字符串和基于路径参数指纹的双重去重策略，可自动标记重复条目并生成冲突报告，避免同一资源在目录中出现多次。

**多格式数据导出** 支持将整理后的链接目录导出为 JSON、YAML 以及 Markdown 表格格式，方便直接嵌入技术文档、项目报告或静态站点生成器中。

**定时扫描与变动通知** 集成了轻量级任务调度模块，能够定期对已入库的链接进行可达性探测（仅检查响应状态码），并将变动结果通过日志文件或自定义钩子函数输出。

**可扩展过滤器链** 提供标准的过滤器接口，开发者可以按需编写自定义过滤插件，实现对特定域名、特定路径模式或特定查询参数的拦截与重写。

## 应用场景

**历史内容平台的数据清洗** 在从老旧 CMS 系统迁移数据至新平台时，原有文章链接通常包含大小写混用、后缀名不规则、多余查询参数等问题。LinkVault Core 可批量导入这些链接，通过路径解析与清洗规则，输出一份标准化的资源清单，供迁移脚本直接使用。

**技术文档库的外链审计** 大型技术文档仓库中常引用大量外部图片、代码示例或参考文章。利用该工具定期扫描文档目录中提取到的所有 URL，可以快速识别失效链接、域名变更条目或非 HTTPS 资源，帮助文档维护者及时修正。

**API 响应数据的链接聚合** 当后端接口返回的 JSON 数据中嵌套了多个指向不同文章详情的 URL 时，开发人员可以使用 LinkVault Core 将这些分散的链接收集起来，统一添加缓存头建议或访问权限标记，再批量下发给前端或第三方调用方。

**运维监控系统的告警关联** 运维团队可将不同监控组件输出的告警详情页链接集中导入 LinkVault Core，再按故障类型、服务模块、严重等级等维度进行标签化整理，最终生成一张可交互的故障处理导航看板。

## 快速开始

以下步骤将帮助您在本地环境中快速启动 LinkVault Core 服务，并完成一次链接导入与导出的完整流程。

```bash
# 克隆项目仓库至本地
git clone https://github.com/your-organization/linkvault-core.git

# 进入项目根目录
cd linkvault-core

# 安装项目依赖（使用 pip 与虚拟环境）
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt

# 运行示例导入任务，处理 sample_links.txt 中的 URL 并输出结果到 output 目录
python cli.py import --input sample_links.txt --output output/ --format json
```

## 安装要求

项目采用 Python 3.9 及以上版本开发，依赖核心库均为开源稳定版本。生产环境建议使用 Linux 操作系统或 WSL2，运行内存不低于 512 MB。下表列出了所有必需的依赖项及其说明。

| 依赖包 | 必需版本 | 说明 |
|---|---|---|

| pip | >= 21.0 | Python 包管理工具，用于安装其余依赖 |
| requests | >= 2.28.0 | 处理 HTTP 请求，用于链接可达性检查与状态探测 |

| click | >= 8.1.0 | 构建命令行交互界面，提供子命令与参数解析能力 |

| black | >= 22.0 | 代码格式化工具，仅在贡献代码时使用 |

## 项目结构

项目采用分层架构设计，核心逻辑与接口层、工具层、数据层分离。以下为项目根目录下的主要目录与文件布局，每行附带简要说明。

```
linkvault-core/
├── cli.py                      # 命令行入口，注册 import、export、scan 等子命令
├── requirements.txt            # 生产环境依赖列表，固定版本号
├── setup.py                    # 项目打包与安装脚本，定义入口点
├── linkvault/                  # 核心代码包
│   ├── __init__.py             # 包初始化，暴露主要接口类
│   ├── engine.py               # 规则引擎主控类，协调解析、清洗与去重流程
│   ├── parser.py               # 路径解析器，实现正则捕获与参数提取
│   ├── filters/                # 过滤器插件目录
│   │   ├── __init__.py
│   │   ├── domain_filter.py    # 域名黑白名单过滤实现
│   │   └── path_filter.py      # 路径模式过滤实现
│   ├── exporters/              # 导出器模块
```

## 常见问题

**Q：导入包含非 ASCII 字符的 URL 时出现编码错误，应如何解决？**

A：LinkVault Core 默认使用 UTF-8 编码读取输入文件。若您的输入文件采用 GBK 或 GB2312 编码，请在调用 import 命令时通过 --encoding 参数显式指定编码，例如 --encoding gbk。同时，建议将输入文件统一转换为 UTF-8 with BOM 格式以避免跨平台兼容性问题。

**Q：如何处理路径中出现的混合大小写后缀，例如 .sHtML 与 .html 同时存在？**

A：项目解析器在提取文件扩展名时默认采用不区分大小写的匹配策略。您可以在 config/default.yaml 中调整 case_sensitive 选项为 false（默认值），此时解析器会将 .sHtML、.SHTML、.shtml 均视为同一种后缀类型。若需保留原始大小写用于其他用途，建议通过扩展字段单独存储 raw_extension 值。

**Q：定时扫描任务是否支持发送邮件或 Webhook 通知？**

A：核心模块本身不包含任何具体通知实现，但提供了 on_status_change 钩子接口。开发者可以继承 BaseNotifier 类并实现 notify 方法，将变动结果推送到企业内部钉钉、飞书、Slack 或自定义 Webhook 地址。具体示例请参考 docs/development.md 中的扩展章节。
