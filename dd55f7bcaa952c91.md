# Vokav Mobile Article Gateway

Vokav Mobile Article Gateway 是一个轻量级的技术文章聚合与导航系统，专为移动端阅读场景设计。该项目将分散在移动站点的技术文章、开发笔记和工程实践文档进行结构化索引，提供统一的访问入口和分类检索能力。项目面向开发者、技术写作者和内容运营人员，帮助其高效管理和分发移动端技术内容资源。

本项目作为第53/56批资源整合节点，对180个移动端文章链接进行系统性梳理和组织，形成可维护、可扩展的技术知识库框架。通过标准化的目录结构和元数据管理，降低内容查找成本，提升技术信息的复用效率。


## 功能概览

**文章ID索引引擎** - 基于文章数字ID建立快速查找映射表，支持通过ID精确命中目标文章，响应时间控制在200ms以内。

**移动端自适应渲染** - 针对手机屏幕尺寸优化内容展示布局，自动调整字体大小与行间距，确保在320px至768px宽度范围内均有良好阅读体验。

**分类标签自动提取** - 从文章URL路径和标题中自动识别技术领域标签，如前端、后端、运维、数据库等，辅助内容归类。

**链接状态健康检查** - 定期对收录的180个链接进行可用性探测，标记失效链接并生成报告，保障资源库的活跃度与可靠性。

**批量导入与导出接口** - 提供JSON和CSV格式的数据导入导出能力，支持与其他内容管理系统进行数据同步。

**访问统计与热度排序** - 记录每篇文章的点击频次和访问时段，按热度对文章列表进行动态排序，突出高频阅读内容。

**全文检索支持** - 基于标题和摘要字段构建倒排索引，支持关键词模糊匹配，帮助用户快速定位相关文章。


## 应用场景

**技术团队内部知识库构建** - 开发团队可将日常遇到的技术问题解决方案、代码调试记录等文章统一收录到本系统中，形成团队共享的技术文档库。通过ID索引和分类标签，成员能够快速回溯历史决策和解决方案。

**移动端技术资讯聚合** - 技术爱好者可使用本系统订阅多个技术博客和新闻源的文章链接，在移动设备上获得统一的阅读列表。系统定期检查链接有效性，避免因源站变动导致的访问失效。

**开源项目文档导航** - 开源项目维护者可将项目相关的设计文档、API参考、升级指南等分散在不同站点的文章通过本系统集中索引，降低新贡献者的学习曲线。支持按版本号或模块名称对文章进行分类组织。

**技术内容运营辅助** - 内容运营人员可利用批量导入接口将外部合作方的技术文章批量录入系统，通过访问统计了解读者偏好，指导后续内容采编方向。导出接口可将数据对接到BI分析工具。


## 快速开始

以下指令帮助您在本地环境中快速启动 Vokav Mobile Article Gateway 服务。

```bash
# 克隆项目仓库
git clone https://github.com/vokav/article-gateway.git
cd article-gateway

# 安装项目依赖（使用 npm）
npm install

# 启动开发服务器，默认监听端口 3000
npm run dev
```

执行上述命令后，在浏览器中访问 http://localhost:3000 即可查看文章网关首页。系统默认加载内置的180条文章索引数据，并自动启动链接健康检查后台任务。

1. 查阅 issues 列表，选择未被指派的 bug 修复或功能增强任务，或在提交新 issue 前搜索是否已有相似提议。建议先从标注为 good-first-issue 的任务入手。

2. Fork 本仓库到您的个人账户，在本地新建功能分支（命名格式为 feature/功能简述 或 fix/问题简述），确保分支从最新的 main 分支切出。

3. 完成代码修改后，运行 npm test 确保所有现有测试用例通过。如新增功能，请同步编写对应的单元测试或集成测试，测试覆盖率不低于原有水平。

4. 提交代码时遵循 Conventional Commits 规范（如 feat: 新增批量导入接口、fix: 修复链接编码解析异常），提交信息需清晰描述变更内容和影响范围。

5. 向 main 分支发起 Pull Request，在 PR 描述中关联对应的 issue 编号，并详细说明变更动机、实现方案和测试结果。至少一名项目维护者审阅通过后方可合并。


## 常见问题

**问：系统启动后无法加载文章列表，页面显示空数据，应如何排查？**

答：首先检查项目根目录下 database/seed/articles.json 文件是否存在且格式合法。该文件包含180条初始文章索引数据，若文件损坏或缺失，系统将无法加载数据。其次确认数据库文件是否有读写权限，SQLite3 在首次启动时会自动创建数据库文件，请确保运行目录可写。若问题持续，可执行 npm run db:reset 强制重置数据库并从种子文件重新导入。

**问：链接健康检查报告显示大量链接超时，是否需要手动干预？**

答：健康检查服务默认采用指数退避重试策略，对超时链接会进行最多3次重试，单次超时阈值为5000毫秒。如报告仍显示大量失败，可能原因包括源站临时不可用、网络环境限制或文章ID对应的资源已被源站移除。建议检查运行环境的网络连通性，并可调整 .env 文件中的 CHECK_TIMEOUT 和 MAX_RETRIES 参数。对于持续失效的链接，可通过管理后台手动标记为废弃状态。

**问：能否将本系统部署到生产环境并支持多用户访问？**

答：本系统当前为单机部署设计，适合小团队内部使用或个人知识库场景。如需支持多用户并发访问，建议在生产环境添加 Nginx 反向代理进行负载均衡，并配置 PM2 或 Docker 实现进程守护。系统默认使用 SQLite3 数据库，在读写并发较高的场景下可考虑迁移至 PostgreSQL，相关适配代码已在 src/models/ 中预留接口。
