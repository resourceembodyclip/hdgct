# LinkVault Core

LinkVault Core 是一个面向技术研究人员、内容聚合者与信息分析团队的高密度外链资源归集系统。本项目不生产原始内容，而是对分布于互联网各节点的既有技术文章、案例文档与参考材料进行结构化整理与统一索引，旨在解决分散信息难以检索、关联性弱、重复采集成本高的问题。目标用户包括技术文档工程师、开源社区维护者、数据分析师以及需要批量访问特定领域历史资料的研究人员。

本项目作为第 41/56 批次发布，当前批次共归集 180 个资源链接，所有链接均指向 wap.mobile.vokav.cn 域名下的技术类文章详情页。LinkVault Core 提供基于 HTTP 协议的资源可访问性校验、链接状态跟踪与基础元数据提取功能，可作为上游数据管道为更高级的文档分析系统提供原始输入。

## 功能概览

**批量链接归集**：支持一次性导入大批量 URL 列表，自动完成去重、格式规范检查与域名白名单过滤，输出标准化的链接集合供后续处理。

**HTTP 状态码探测**：对每个资源链接执行异步 HEAD 请求，记录响应状态码、内容类型与内容长度，快速识别失效或重定向链接。

**元数据自动提取**：从目标页面 HTML 中抽取标题、段落首句与最后修改时间戳，生成可用于建立简易搜索索引的结构化记录。

**链接生命周期跟踪**：记录每次采集任务的执行时间、成功数与失败数，支持按批次维度查询历史采集结果，便于监控源站内容变动。

**增量更新机制**：支持基于已有批次 ID 执行增量采集，仅对新增或变更的链接重新抓取，避免全量重复处理。

**原始数据导出**：提供 JSON Lines 与 CSV 两种格式的导出接口，便于下游系统进行二次处理或导入数据库。

## 应用场景

**技术文档归档前质量检查**：团队在将第三方技术文章归档至内部知识库之前，可通过 LinkVault Core 批量验证所有引用链接的可达性，提前发现并标记已失效的来源，避免归档内容出现断链。

**开源项目依赖文档追溯**：开源项目维护者可使用本系统批量采集项目 README 或官网中引用的外部技术参考链接，定期检查这些资源是否仍然在线，确保项目文档中外部引用的长期有效性。

**历史数据迁移前资源盘点**：在进行网站改版或数据迁移时，运维人员可借助本系统对旧站点中引用的全部外链生成清单，评估迁移后重定向规则的覆盖范围，降低因链接变更导致的流量损失。

## 快速开始

以下操作步骤帮助您在本地环境中快速启动 LinkVault Core 服务。

```bash
# 克隆项目仓库至本地
git clone https://github.com/linkvault/core.git

# 进入项目根目录
cd core

# 安装项目依赖（使用 pip 与 requirements.txt）
pip install -r requirements.txt

# 运行核心采集模块，导入当前批次资源列表
python run.py --batch 41 --input ./data/batch_41_urls.txt --output ./output/batch_41_result.json
```

## 项目结构

项目目录按照功能模块进行组织，核心采集逻辑与工具函数分离。

```
core/
├── run.py                      # 项目入口脚本，解析命令行参数并调度采集流程
├── requirements.txt            # 项目依赖声明文件，列出所有必需的第三方库及版本
├── config/
│   ├── settings.py             # 全局配置定义，包含超时时间、重试策略与并发数
│   └── logging.conf            # 日志格式与输出级别配置
├── src/
│   ├── __init__.py             # 包初始化文件
│   ├── fetcher.py              # HTTP 请求封装，执行链接探测与重定向跟踪
│   ├── parser.py               # HTML 解析模块，提取标题、段落与时间戳
│   ├── validator.py            # URL 格式校验与域名白名单过滤逻辑
│   ├── batch.py                # 批次管理模块，处理批次创建、查询与状态更新
│   └── exporter.py             # 结果导出模块，支持 JSON Lines 与 CSV 格式
├── tests/
│   ├── test_fetcher.py         # fetcher 模块的单元测试用例
│   ├── test_parser.py          # parser 模块的单元测试用例
│   └── test_batch.py           # batch 模块的单元测试用例
├── data/
│   ├── batch_41_urls.txt       # 当前批次输入的原始链接列表文件
│   └── history/                # 历史批次采集结果存档目录
├── output/
│   └── batch_41_result.json    # 当前批次采集输出的结果文件
└── docs/
    ├── user_guide.md           # 用户手册
    ├── config_reference.md     # 配置参数参考文档
    ├── api.md                  # 模块接口文档
    └── batch_management.md     # 批次管理操作指南
```

## 贡献指南

我们欢迎社区贡献者参与 LinkVault Core 的开发与改进。请遵循以下流程提交您的贡献。

1. 在 GitHub 仓库的 Issues 区域查找标记为 "help wanted" 或 "good first issue" 的工单，或提交新的 Issue 描述您希望解决的问题或新增的功能。

2. 从主分支检出新的功能分支，分支命名遵循 `feature/描述` 或 `fix/描述` 的格式，确保代码风格与项目现有规范一致，所有新增功能需附带对应的单元测试。

## 常见问题

**问：采集过程中遇到 HTTP 403 或 429 状态码时如何处理？**

答：LinkVault Core 内置了指数退避重试策略。当遇到 429（请求过多）时，系统会根据响应头中的 Retry-After 字段或默认等待时间自动延迟后重试。若遇到 403（禁止访问），建议检查目标站点是否要求特定的 User-Agent 或 Referer 头，您可以在 config/settings.py 中自定义请求头部。若重试 3 次后仍失败，系统会将该链接标记为最终失败并跳过。

**问：如何更新已存在批次中的链接状态而不重新采集全部数据？**

答：您可以使用增量更新模式。在命令行中指定 --batch 参数为已有批次号，并添加 --incremental 标志。系统会重新请求该批次中所有链接，但仅当页面内容发生变更（通过对比响应内容长度或最后修改时间）时才会更新存储记录，未变更的链接将保留原状态。这一模式可显著降低不必要的网络请求。

**问：导出的结果文件中包含哪些字段？**

答：导出的 JSON Lines 文件中每行对应一个链接的采集结果，包含以下核心字段：url（原始链接）、status_code（HTTP 响应状态码）、content_type（返回内容的 MIME 类型）、title（页面标题）、first_paragraph（正文首段文本）、last_modified（页面最后修改时间，若存在）、fetch_timestamp（采集完成的时间戳）以及 error_message（若失败，记录错误原因）。CSV 格式包含相同的列。
