# LinkVault Core

LinkVault Core 是一个面向技术团队与内容运营者的轻量级外链资源聚合与管理平台。该项目定位于解决分散在多处的技术文档、教程、工具站与内部知识库链接难以统一检索、版本跟踪与可用性监控的问题。LinkVault Core 不提供内容存储，而是作为链接的元数据管理中心，支持标签体系、访问状态检测与批量导入导出，适用于中小型开发团队、技术内容站点以及个人知识管理场景。通过结构化的链接组织方式，项目帮助用户将零散的书签与收藏转化为可维护、可共享、可观测的技术资源资产。

## 功能概览

**批量链接导入与解析** 支持从文本文件、CSV 及主流浏览器书签导出文件中批量导入链接，自动解析标题、域名及路径信息，并识别重复条目。

**链接可用性主动检测** 内置定时检测任务，对存储的所有链接进行 HTTP 状态码与响应时间检查，标记失效或超时链接，支持告警通知。

**多级标签与分类体系** 允许用户为每个链接赋予多个自定义标签，并通过标签组合进行快速筛选，支持创建嵌套分类目录。

**全文检索与高级过滤** 基于链接标题、URL 片段、描述及标签的全文检索功能，配合状态、域名、时间范围等多维度过滤器。

**链接变更历史追踪** 记录链接标题、描述或目标 URL 的每一次修改，支持回滚至任意历史版本，便于审计与内容回溯。

**数据导入导出与迁移** 支持将全量链接数据导出为 JSON、CSV 及 HTML 书签格式，同时支持从上述格式恢复数据，便于跨系统迁移。

**访问统计分析** 提供链接的点击次数、最后访问时间、检测通过率等统计指标，辅助识别高频使用与长期未访问的资源。

**团队协作与权限控制** 支持多用户工作区模式，可分配只读、编辑与管理三种角色，实现团队内链接资源的共享与分级管理。

## 应用场景

技术团队内部知识库维护。开发团队可将项目文档链接、设计规范、API 参考与运维手册统一收录至 LinkVault Core，通过标签区分环境或模块，结合状态检测及时发现已迁移或下线的内部页面，减少因链接失效导致的信息获取阻塞。

技术博客与内容站点外链管理。内容创作者在文章中引用大量外部参考链接时，可使用 LinkVault Core 集中管理这些引用源，定期检测外链可用性，在链接失效前主动替换或更新，提升站点内容的长期可读性。

个人技术学习资源沉淀。技术人员可将日常阅读的教程、工具库、论文与视频课程链接按技术栈或学习阶段分类存储，利用检索与过滤功能快速定位特定主题资料，避免重复搜索，提升学习效率。

开源项目依赖与生态资源索引。开源维护者可将项目依赖的文档站、社区论坛、CI/CD 服务面板及镜像源地址统一管理，作为项目 README 或贡献指南的补充资源库，降低新贡献者的环境搭建与信息查找成本。

## 快速开始

以下步骤将指导您在本地环境快速启动 LinkVault Core 服务。

```bash
# 克隆项目仓库
git clone https://github.com/linkvault/linkvault-core.git

# 进入项目目录
cd linkvault-core

# 安装项目依赖（使用 npm）
npm install

# 配置环境变量，复制示例配置并修改数据库连接等信息
cp .env.example .env

# 执行数据库迁移与初始数据填充
npm run migrate
npm run seed

# 以开发模式启动服务（默认监听端口 3000）
npm run dev
```

启动成功后，访问 http://localhost:3000 即可进入 LinkVault Core 的 Web 管理界面。首次访问将引导创建管理员账户与初始工作区。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---|---|---|
| Node.js | 18.x 或 20.x LTS | 项目运行时环境，建议使用最新的 LTS 版本以获取长期安全更新 |
| PostgreSQL | 14.x 及以上 | 主数据库，用于存储链接元数据、标签关系与用户信息 |
| Redis | 6.x 及以上 | 缓存与任务队列后端，用于提升查询性能及调度检测任务 |
| npm 或 yarn | 对应 Node.js 18/20 的配套版本 | 包管理器，用于安装项目依赖与执行脚本命令 |
| Git | 2.x 及以上 | 版本控制工具，用于克隆仓库及管理代码更新 |

## 项目结构

```
linkvault-core/
├── src/                           # 核心源代码目录
│   ├── api/                       # RESTful API 路由与控制器层
│   │   ├── routes/                # 各模块路由定义（links, tags, users, checks）
│   │   └── middlewares/           # 鉴权、日志、限流等中间件
│   ├── core/                      # 核心业务逻辑层
│   │   ├── link-manager/          # 链接增删改查、标签关联、历史记录服务
│   │   ├── health-checker/        # 链接可用性检测调度与执行引擎
│   │   └── import-export/         # 批量导入导出处理器（支持 CSV/JSON/HTML）
│   ├── models/                    # 数据模型与数据库映射（使用 ORM）
│   │   ├── link.model.js          # 链接实体模型定义
│   │   ├── tag.model.js           # 标签实体模型
│   │   └── user.model.js          # 用户与工作区权限模型
│   ├── services/                  # 外部服务集成层
│   │   ├── cache/                 # Redis 缓存封装服务
│   │   ├── queue/                 # 基于 Bull 的任务队列服务
│   │   └── notifier/              # 邮件与 Webhook 通知服务
│   └── utils/                     # 通用工具函数集合
│       ├── validators/            # URL 格式校验、标签名合规性校验
│       └── parsers/               # 书签文件解析、HTML 元数据提取工具
├── config/                        # 环境配置文件与常量定义
│   ├── default.js                 # 默认配置参数（端口、数据库连接池等）
│   └── production.js              # 生产环境覆盖配置
├── migrations/                    # 数据库结构迁移脚本（按版本号组织）
├── seeders/                       # 初始测试数据与预置标签种子文件
```

## 常见问题

Q: 检测链接可用性时，系统是否会频繁访问外部站点，造成被检测服务器的负担？

A: LinkVault Core 的检测模块默认采用渐进式间隔策略，同一域名下的链接会共享检测队列，避免短时间内对同一服务器发起大量并发请求。用户可在配置文件中调整并发度（concurrency）与超时时间（timeout），对于内部或敏感站点，可设置检测白名单或完全关闭自动检测，仅使用手动触发模式。

Q: 项目支持从浏览器书签直接导入链接吗？具体支持哪些浏览器的书签格式？

A: 支持。系统内置了适用于 Chrome、Firefox 和 Edge 浏览器的书签 HTML 导出文件的解析器。您只需从浏览器书签管理器中导出为 HTML 文件，然后在 LinkVault Core 的导入界面中选择该文件上传即可。系统会自动解析书签文件夹结构并转换为标签体系，同时保留书签的原始添加时间与标题。

Q: 如果我希望将 LinkVault Core 部署在团队内部网络，且无法连接外部互联网，是否会影响核心功能？

A: 不会影响核心的链接管理与检索功能。唯一受影响的是链接可用性检测模块中对外部 CDN 或公共解析服务的依赖——系统默认会尝试使用公共 DNS 进行域名解析，您可以将检测模块的 dns 配置项替换为内网 DNS 服务器地址。此外，导入功能中的自动元数据抓取（如页面标题、描述）在无外网环境下不可用，但您仍可手动编辑这些字段。
