# ResourceLink Navigator

ResourceLink Navigator 是一个面向技术开发者和信息研究人员的结构化外链资源导航系统。该项目并非一个传统的网站应用，而是一个经过精心分类、索引和标注的综合性技术文档与资讯链接集合。项目通过规范化的 Markdown 文档对分散于网络的高价值技术文章、教程和案例进行重新组织，解决了开发者面对海量信息时检索效率低下、优质内容难以沉淀的痛点。

本项目定位为“技术资源的索引之索引”，目标用户包括运维工程师、后端开发者、移动端架构师以及技术决策者。项目组定期对收录的 URL 进行可用性校验与分类更新，确保每一项资源均能稳定访问并具有明确的技术指向性。通过本导航系统，用户可以在数秒内定位到特定框架的实践案例、性能调优记录或架构设计复盘，显著降低信息过滤的时间成本。

## 功能概览

- **智能资源聚合**：系统将来自同一站点的数千篇技术文档按主题自动聚类，用户可依据文章标题前缀或 ID 规律快速判断内容所属技术栈。

- **多维度检索过滤**：提供基于文章 ID、发布时间、所属子目录的复合筛选逻辑，支持用户在庞大链接库中精确定位目标文档。

- **状态监控面板**：集成链接存活检测机制，对返回 HTTP 异常状态的资源进行标记，并在文档中生成可视化的健康度报告。

- **离线缓存索引**：生成资源列表的本地快照文件，允许用户在没有网络连接的情况下查阅已收录资源的元数据信息。

- **版本对照视图**：针对同一技术主题下的多篇关联文章，自动建立“前置阅读”与“延伸阅读”的交叉引用关系，形成知识图谱。

- **自定义标签系统**：用户可为任意链接添加私有标签，并通过标签云进行个性化分类管理，满足团队内部知识库共建需求。

- **导入导出接口**：支持将当前资源列表以 JSON、CSV 或纯文本格式导出，便于迁移至其他文档管理平台或进行离线数据分析。

- **变更追踪日志**：记录每次资源列表更新的操作时间、添加条目数和失效链接数，为项目维护提供完整的审计追溯能力。

## 应用场景

- **技术选型前的调研阶段**：当架构团队需要评估某一中间件或数据库在真实生产环境中的表现时，可通过本导航系统快速检索相关性能测试报告与故障复盘文章，避免重复造轮子。

- **新员工技术培训路径规划**：团队负责人可以将本资源列表中的特定文章序列整理为培训大纲，新人根据索引顺序依次阅读，能够在较短时间内建立起对核心业务系统的认知框架。

- **线上故障应急响应**：运维人员在处理突发宕机或性能劣化时，可通过本系统查询历史相似案例的处理记录，借鉴前人的解决方案，从而缩短平均修复时间。

- **技术博客与文档撰写参考**：技术作者在撰写深度技术文章时，可以利用本导航系统中的参考文献作为论据支撑，提升内容的技术深度和可信度。

- **开源社区贡献者 onboarding**：开源项目维护者可以将本系统中的相关链接作为贡献者入门指南的补充材料，帮助新手理解项目周边的生态工具与部署规范。

## 快速开始

以下步骤将指导您在本机搭建 ResourceLink Navigator 的离线文档浏览环境。

```bash
# 克隆项目仓库至本地
git clone https://github.com/resourcelink/navigator.git

# 进入项目根目录
cd navigator

# 安装依赖（项目基于 Node.js 18+ 与 Python 3.10+ 构建文档工具链）
npm install -g markdown-link-check
pip install mkdocs mkdocs-material

# 构建静态站点并启动本地预览服务
mkdocs build
mkdocs serve
```

执行上述命令后，在浏览器中访问 `http://127.0.0.1:8000` 即可查看完整的资源导航界面。若您只需要纯 Markdown 格式的资源列表，可直接打开项目根目录下的 `RESOURCES.md` 文件。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---|---|---|
| Node.js | 18.x LTS 或更高 | 用于运行链接有效性检查脚本与 Markdown 解析工具 |
| Python | 3.10 至 3.12 | MkDocs 静态站点生成器的运行环境，同时用于自定义插件开发 |
| mkdocs-material | 9.x | 项目使用的 Material 主题，提供响应式布局与搜索功能 |
| markdown-link-check | 3.x | 全局安装的 NPM 包，用于批量校验文档内外部链接的可达性 |
| Git | 2.30 或更高 | 用于克隆仓库、管理版本分支及提交贡献变更 |
| 操作系统 | Linux / macOS / Windows WSL2 | 项目构建脚本在 Unix-like 环境下测试通过，Windows 用户建议使用 WSL2 |
| 网络带宽 | 不低于 1 Mbps | 初次构建时需从 CDN 下载主题资源，且链接校验过程需要访问外网 |
| 磁盘空间 | 至少 200 MB | 用于存放构建产物、缓存文件及本地索引数据库 |

## 项目结构

项目采用分层目录结构，将源码、文档、配置和构建产物严格分离，便于长期维护和自动化部署。

```
navigator/
├── docs/                                 # 文档源文件目录，所有 Markdown 内容存放于此
│   ├── index.md                          # 站点首页，展示项目概述与快速导航入口
│   ├── RESOURCES.md                      # 核心资源列表，包含全部 180 个链接的完整清单
│   ├── REPORT.md                         # 链接健康度报告，每 24 小时自动更新一次
│   ├── TAGS.md                           # 标签索引文件，按技术主题分类组织资源
│   ├── CONTRIBUTING.md                   # 贡献者指引，详细说明新增链接的提交流程
│   └── CHANGELOG.md                      # 版本变更记录，按批次号追溯每次更新内容
├── scripts/                              # 工具脚本目录，用于自动化维护与校验
│   ├── check_links.js                    # Node.js 脚本，批量检查所有 URL 的 HTTP 状态
│   ├── generate_report.py               # Python 脚本，根据检查结果生成 REPORT.md
│   ├── sort_by_id.py                    # 按文章 ID 数值排序资源列表的辅助工具
│   └── validate_format.sh               # Shell 脚本，校验 URL 格式是否符合规范要求
```

## 贡献指南

我们欢迎所有技术爱好者参与 ResourceLink Navigator 的资源扩充与维护工作。请遵循以下标准流程提交贡献。

1.  **复刻仓库并创建特性分支**：从主仓库复刻代码库至您的个人账户，然后基于 `main` 分支创建新的特性分支，分支命名格式为 `feature/add-resources-batch-xx`。

2.  **编辑资源列表文件**：在 `docs/RESOURCES.md` 文件中追加新的链接条目。每个链接必须独占一行，并确保 URL 格式与现有条目保持一致（包含协议、域名、路径及文件扩展名）。新增链接请放置于对应 ID 区间的末尾。

3.  **本地验证链接可用性**：在提交前，请运行 `scripts/check_links.js` 脚本对新增的链接进行有效性检查。确保所有新增 URL 均返回 HTTP 200 状态码。如存在重定向，请更新为最终目标 URL。

4.  **更新变更日志**：在 `docs/CHANGELOG.md` 中记录本次新增的链接数量、涉及的主题分类以及任何移除或替换的链接。使用当前日期作为版本批次号。

5.  **发起拉取请求**：将您的特性分支推送至复刻仓库，并向主仓库发起拉取请求。在请求描述中清晰说明本次新增资源的技术领域和选择依据。项目维护者将在 48 小时内进行评审与合并。

## 常见问题

**问：部分链接访问时返回 404 或超时错误，我应该如何处理？**

答：项目维护团队已建立每日自动检查机制，并在 `REPORT.md` 中标注异常链接。如果您发现某个链接在报告中未被标记但实际无法访问，请提交 Issue 说明具体情况。对于长期失效的链接，团队将定期进行移除或替换，以确保资源列表的可用性。

**问：我可以提交其他域名下的链接吗？当前项目是否只收录 `mobile.vokav.cn` 这一站点的内容？**

答：当前批次（第 17 批）的资源统一来自 `mobile.vokav.cn` 域，这是为了便于项目初期的规范化管理。但项目长期规划是建立跨域的技术资源聚合平台。如果您推荐其他高质量技术站点的文章，可以在贡献时附加说明，维护团队评估后会将新域名纳入收录白名单。

**问：如何获取项目的最新更新或接收链接状态变更通知？**

答：建议您 Watch 本项目的 GitHub 仓库，选择“Custom”并勾选“Releases”和“Pull requests”事件。每当 `CHANGELOG.md` 或 `REPORT.md` 发生更新时，您将收到邮件通知。此外，您也可以订阅项目的 RSS 源（位于 `/feed.xml`）以获取文档变更摘要。
